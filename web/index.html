Project overview

ESCO is a European taxonomy that links occupations to skills and competences. Here we focus on an AI-related subset (occupations + skills) and use ESCO relations to study how AI competence is represented: what is defined as essential (core requirements) and what is defined as optional (peripheral specialization).

Main research question: How are AI-related occupations structured in terms of core (essential) and peripheral (optional) competences in the ESCO taxonomy?

Problem

ESCO lists many AI-related occupations and skills, but it is not immediately clear which competences are shared across roles and which are role-specific.

Approach

We analyse occupation–skill links as a bipartite structure and use interactive views to compare core vs optional skills, shared skills, clusters, and bridges.

Reading guide (storyline): we start from how many skills each occupation has (Section 1), then we ask which skills are most shared (Section 2), we add taxonomy context by positioning roles in the ISCO hierarchy (Section 3), and finally we show structure (clusters, niche skills, bridges) using a matrix and a network view (Sections 4–5).
Important note: this is a taxonomy-level analysis. Counts reflect ESCO modelling choices (which links exist, and how they are labelled), not labour-market demand, frequency, or hiring trends.
Data scope: in the visual analysis we keep only occupations linked to at least 2 AI-related skills (preprocessing threshold), to remove trivial/noisy cases.

1. How large is the AI skill core for each occupation?

First, we examine the balance between essential and optional AI-related skills for each occupation. This addresses a simple but important question: does ESCO describe some roles as requiring a large AI “core”, while others rely mainly on a small essential set complemented by optional skills?

Why this matters. The essential/optional distinction in ESCO reflects the difference between baseline requirements and role-specific enrichment. Large imbalances in this split across occupations therefore suggest different degrees of standardisation versus specialisation in how AI-related competences are defined.

How to read: each bar = one occupation. Segments show counts of essential vs optional AI skills. What you can do: hover to inspect; click a segment to pin the highlight for comparison; click on empty space to reset. Reminder: values represent ESCO links (taxonomy modelling), not real-world demand.

Takeaway: Loading summary…

2. Which AI skills form a shared core across occupations?

After comparing occupations, we switch perspective and rank skills by how many AI-related occupations they appear in. Skills that recur across many occupations represent a shared “baseline” in ESCO. We also distinguish whether a skill is labelled as knowledge (understanding) or skill/competence (doing).

What to look for: do the most common AI competences look mainly conceptual (knowledge) or operational (skill/competence)? A mixed top suggests that ESCO models AI competence as both foundations + practice.

What you can do: hover to inspect a skill; use filtering (if available) to compare knowledge vs skill/competence. If the chart supports pinning: click an element to keep it highlighted; click outside to clear.

Takeaway: A small set of skills is shared across many occupations, suggesting a common AI backbone in the taxonomy. Less frequent skills tend to mark specialization or niche role requirements.

3. ISCO context: where do AI-related occupations sit?

So far, we treated “AI-related occupations” as a single set. Here we add taxonomic context by placing them inside the ISCO hierarchy (major groups → sub-groups → occupations). This answers a different question: is AI in ESCO concentrated in a few professional families, or spread across multiple parts of the labour taxonomy?

Interpretation tip: bars represent groups in ISCO, and their values are the sum of AI-related skill links across all AI-related occupations contained in that group. This is not “how many people work there”, nor a demand signal — it is how ESCO models AI competence across ISCO categories.

How to read: each bar is an ISCO group within the current view. The bar length encodes the total number of AI-related occupation–skill links in that group’s subtree. What you can do: click a bar to drill down to finer ISCO levels; use the breadcrumb to jump back to any ancestor; use Back or Reset view to navigate quickly. Why it matters: the drilldown reveals where ESCO places AI work: concentration suggests a “professional core”, while dispersion signals cross-domain uptake.

Takeaway: AI-related occupations cluster strongly within professional and technical groups, with much smaller footprints elsewhere. Drilling down highlights which sub-domains (e.g., science/engineering vs ICT vs business) carry most of the taxonomy’s AI links.

Design choice: colours encode the ISCO major group (top-level ancestor). As you drill down, the colour stays stable to preserve orientation and avoid re-learning the palette at every level.

4. Patterns of shared vs specialized competences

Rankings are useful, but they hide structure. This matrix view makes patterns explicit: which occupations share similar AI skill sets (clusters), and which skills appear only in a few roles (specialization). With ordering controls we can compare a “totals-first” view against a similarity-oriented arrangement.

How to explore: try switching between Default and Cluster ordering. Then use Search to check whether a set of occupations (e.g., “marine”, “chief”, “envi”) forms a coherent pattern. Finally, pin a skill (column) to see exactly which occupations include it.

Encoding: rows = occupations, columns = skills; a filled cell indicates that the occupation is linked to that skill in ESCO. What you can do: hover a cell to see details; use Search to highlight matching rows/columns; click a column number (or the legend item) to pin / unpin that skill and isolate its footprint; click outside to clear.

Takeaway: Clusters reveal families of occupations that share a common competence backbone. Sparse or isolated columns indicate niche skills that likely represent specialization in ESCO.

5. Occupation–skill network: clusters and bridges

The matrix makes co-occurrence visible, but a network view is better to reason about connectivity: which skills connect otherwise separate groups of occupations, and which occupations sit at the boundary between clusters. This supports an interpretation in terms of transferability of competences.

What to look for: skills with many connections can act as “bridges” between groups. If your implementation supports interaction, use hover (and zoom/pan) to inspect who connects to whom.
Network view

Tip: hover to see neighbors; click to pin a node; search to highlight matches; drag nodes to explore.

Goal: highlight clusters and bridge skills that link multiple occupations. Interaction: hover nodes/links for details; zoom/pan if available.

Takeaway: Bridge skills connect clusters of occupations, suggesting where competence transfer might be easier (at least from a taxonomy perspective).

Methodology & reproducibility

This section documents data sources, preprocessing, definitions, and the files used by each visualization to make the project reproducible end-to-end.

Data sources

  • ESCO occupations and skills (European Commission): concept schemes and occupation–skill relationships, focusing on AI-related concepts.
  • ISCO (International Standard Classification of Occupations): used to provide a hierarchical drilldown structure for occupations.

Preprocessing pipeline

We convert raw ESCO relations into analysis-ready tables for D3 via: cleaning (label normalization, duplicates removal), filtering (AI-related concepts), and aggregation (per occupation, per skill, and per ISCO subtree). The preprocessing code (Python scripts/notebooks) is included in the repository and can be rerun to reproduce all derived datasets.

Operational definitions

  • AI-related skill link (edge): one ESCO relation connecting an occupation to an AI-related skill. Type: essential or optional.
  • Occupation total: number of AI-related skill links (essential + optional) for an occupation.
  • Skill frequency: number of occupations linked to a skill.
  • ISCO subtree total: sum of AI-related skill links across descendant occupations (hierarchy summation).
  • Matrix cell: binary presence in the selected subset (1 = link exists, 0 = absent/not shown).

Reproducibility

The public GitHub repository includes: (1) preprocessing scripts, (2) the derived datasets used by the website, and (3) a README.md explaining how to rerun preprocessing and serve the website locally.
